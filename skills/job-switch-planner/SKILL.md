# Job Switch Planner

## Overview

引导技术岗跳槽者完成从自我认知到落地的完整 7 阶段流程：用户画像 → 期望收集 → 公众评价 → 前景分析 → 岗位确定 → 招聘信息采集 → 跳槽计划。每个阶段产出检查点，支持回退重算。

## 使用时机

- 用户说"我想跳槽"、"帮我分析一下该往哪个方向走"
- 用户想了解某个岗位的市场前景
- 用户需要制定系统的跳槽/学习计划
- 用户想在 Boss 直聘/猎聘等平台看某个岗位的要求

## 核心架构

### 状态管线

```
CP1(画像) → CP2(期望) → CP3(公众评价) → CP4(前景分析) → CP5(岗位确定) → CP6(招聘信息) → CP7(跳槽计划)
```

- 顺序执行，每阶段产出检查点
- 用户可随时回退到任意检查点，修改后后续阶段重新计算
- 所有检查点数据存储在 `job-switch-planner/user_journey.json`（相对于当前工作目录）

### 交互模式

对话引导 + 关键节点展示分析结果 → 用户确认/调整 → 继续下一阶段

### 技术依赖

- Playwright：自动化采集知乎、脉脉、V2EX、Boss直聘等公开信息
- 无需外部 API
- 最终报告输出为 Markdown

## 阶段流程

### CP1：用户画像

**目标：** 收集用户的背景信息，建立结构化画像。

**输入方式：** 优先让用户粘贴简历（文本），也可以对话逐项问答。

**引导话术模板：**
> "请粘贴你的简历内容，或者我们逐项聊聊你的背景。先从技术栈开始吧——你主要用什么语言、框架和工具？"

**采集字段：**

```yaml
姓名/昵称:   # 用于报告生成和文件命名
技术栈:
  语言: []
  框架: []
  工具: []
工作年限:
当前职级:
当前薪资:
教育背景:
  学历:
  专业:
  院校:
项目经验:
  - 名称:
    角色:
    关键词: []
软技能: []
```

**缺失字段处理：** 自动识别缺失项，主动追问。追问时只问缺失的字段，不要重复已有信息。

**项目经验解析指引：** 如果用户粘贴简历，按以下流程解析：简历文本 → 按公司/项目分段 → 提取项目名称、角色、技术关键词。如信息不足则追问。

**产出检查点：** 将结构化数据保存为 `job-switch-planner/user_journey.json` 的 `profile` 字段。

### CP2：期望收集

**目标：** 了解用户对新工作的期望。

**引导话术模板：**
> "了解了你的背景，现在聊聊你对下一份工作的期待吧。你想找什么方向的岗位？期望薪资范围大概是多少？"

**采集字段：**

```yaml
目标岗位: []
目标行业: []
期望职级:
目标技术栈:
  想继续用的: []
  想转型的: []
  不想再碰的: []
期望薪资:
工作地点偏好: []
公司规模偏好: []
最看重的因素: []
```

**技能行为：**
- 基于画像给出建议（如"你 5 年后端经验，市场通常对应 P6-P7 级别"）
- 如果用户期望明显不合理（如 3 年经验期望薪资 100K），温和提示市场行情
- **方向不匹配检测：** 如果用户目标岗位与当前技术栈所在职能领域差异极大（如前端转 AI 算法），提示转型成本和可行路径："这两个方向的技术栈交集很小，转型通常需要 X 个月的系统学习，你确定要走这个方向吗？"
- **参考区间来源：** CP2 阶段的薪资参考来自 LLM 自身知识。如果用户需要更精确的数据，告知"后续 CP6 我会帮你采集真实的招聘薪资数据"
- **复合异常处理：** 当多个不合理因素同时出现时，逐一说明每个问题，最后给出总体建议

**产出检查点：** 保存为 `job-switch-planner/user_journey.json` 的 `expectations` 字段。

### CP3：公众评价

**目标：** 在知乎、脉脉、V2EX、掘金等平台采集关于目标岗位的讨论。

**采集方式：** 使用 OpenCLI 进行搜索（参见 [信息采集](#信息采集基于-opencli)）

**搜索关键词：** 使用 CP2 中用户填写的 `目标岗位`，格式如 `"{目标岗位} 前景"`、`"{目标岗位} 薪资"`、`"{目标岗位} 裁员"`
**多目标岗位处理：** 如果用户选了多个目标岗位，优先搜索匹配度最高的那个；用户也可指定要分析的方向
**每平台采集：** 前 3-5 条相关帖子的标题和摘要内容

**输出结构：**

```yaml
public_opinion:
  - 来源: "知乎"
    标题: "Go 后端开发前景怎么样？"
    摘要: "..."
    倾向: "正面/负面/中性"
  - 来源: "脉脉"
    标题: "..."
    摘要: "..."
    倾向: "中性"
```

**交互：** 采集完成后展示摘要给用户确认。用户可追问某个来源的详情。

**产出检查点：** 保存为 `job-switch-planner/user_journey.json` 的 `public_opinion` 字段。

### CP4：前景分析

**目标：** 基于 CP3 采集的数据，分析目标岗位的市场前景。

**分析方法：**
- 归纳各平台讨论的主题方向（如"技术前景"、"薪资待遇"、"裁员风险"）
- 提取薪资关键词 → 薪资范围（标注数据来源可靠性：招聘帖较可靠，讨论帖仅供参考）
- 提取槽点和正面信号
- 综合给出整体评价
- **注意：** 3-5 条帖子的样本量不足以判断"热度趋势"，所以不做趋势断言，只做主题归纳

**输出结构：**

```yaml
market_analysis:
  岗位: "Go 后端工程师"
  热度趋势: "平稳上升/下降/波动"
  薪资范围: "35K-60K"
  常见槽点: []
  正面信号: []
  整体评价: "..."
  信息来源: ["知乎 N帖", "脉脉 N帖", ...]
```

**交互：** 展示分析结果，用户可追问详情。阶段结束后提示进入下一阶段。

**产出检查点：** 保存为 `job-switch-planner/user_journey.json` 的 `market_analysis` 字段。

### CP5：岗位确定

**目标：** 综合画像 + 期望 + 市场分析，推荐目标岗位让用户选择。

**推荐逻辑：** 多维度加权评分，每个维度 0-100 分，最终加权平均。

**评分维度与权重：**

| 维度 | 分值范围 | 权重 | 计算方式 |
|------|---------|------|---------|
| 技术栈匹配度 | 0-100 | 35% | 目标岗位要求 vs 用户画像技术栈，Jaccard 相似度。零交集时（如前端 vs 算法）直接给 0 分并标注"方向鸿沟" |
| 工作年限匹配 | 0-100 | 15% | 对照目标岗位典型要求（如"3-5年"取中值），差距每多 1 年扣 15 分 |
| 职级匹配 | 0-100 | 10% | 当前职级 vs 目标岗位常见职级，同级 100，每差一级扣 25 分 |
| 学历匹配 | 0-100 | 5% | 目标岗位普遍要求 vs 用户学历。要求匹配 100，高一级 80，低一级 40，低两级 0 |
| 市场前景 | 0-100 | 20% | 基于 CP4 分析：前景好 80-100，平稳 50-79，下行 <50 |
| 期望薪资匹配 | 0-100 | 10% | 期望薪资中值 vs 市场薪资范围中值，偏差 <20% = 100，每多 10% 扣 10 分 |
| 地点匹配 | 0-100 | 5% | 用户偏好城市 vs 岗位主要聚集城市，完全匹配 100，部分匹配 60 |
| 行业经验迁移 | 0-100 | 5% | 用户当前行业 vs 目标行业关联度，同行业 100，相关行业 70，跨行业 40 |

**权重调整规则：** 如果用户在 CP2 的"最看重的因素"中勾选了特定项，对应维度权重上调 50%（如勾选"薪资涨幅"，薪资匹配权重从 10% 变为 15%）。其他维度按比例压缩：`调整后权重 = 原始权重 × (1 - 总上调量) / (1 - 上调维度的原始权重和)`，确保所有权重之和保持 100%。

**最终匹配度公式：**
```
匹配度 = Σ(维度分 × 维度权重) / Σ(维度权重)
```

**推荐筛选与分级话术：**

| 匹配度 | 处理方式 |
|-------|---------|
| ≥ 80% | 强烈推荐，标注为"高度匹配" |
| 60-79% | 推荐，标注为"有潜力" |
| 40-59% | 作为参考，标注为"需努力"，逐一列出差距项 |
| < 40% | 温和劝阻，说明为什么不推荐："这个方向与你的背景匹配度很低（主要原因是 X、Y、Z），如果你还是想尝试，建议先做针对性的学习" |

**技术栈零交集处理：** 当 Jaccard 相似度为 0 时（如前端→算法），匹配度直接 ≤ 40%，按上表给出强劝阻话术。

**输出结构：**

```yaml
recommendations:
  - 岗位: "云原生 Go 后端工程师"
    匹配度: 92%
    匹配理由: "..."
    风险: "..."
```

**交互：**
1. 展示推荐列表给用户
2. 用户选择最终目标岗位（可多选）
3. 用户也可手动输入自己的选择
4. 用户确认后进入下一阶段

**产出检查点：** 保存为 `job-switch-planner/user_journey.json` 的 `recommendations` 和 `selected_positions` 字段。

### CP6：招聘信息采集

**目标：** 用 Playwright 在 Boss 直聘、猎聘等平台采集用户选定岗位的最近招聘信息。

**采集方式：** 使用 OpenCLI 搜索职位（参见 [信息采集](#信息采集基于-opencli)）

**搜索关键词：** 用户选定的岗位名称
**每平台采集：** 最近 30 条职位
**采集内容：** 岗位要求、福利待遇、薪资范围、公司名称
**后处理：** 提取高频词分析市场需求

**输出结构：**

```yaml
job_listings:
  岗位: "云原生 Go 后端工程师"
  来源: "Boss直聘"
  样本量: 30
  requirements_高频词:
    - K8s (28/30)
    - 微服务 (24/30)
  gap_analysis:
    满足: []
    需补充: []
```

**gap_analysis 逻辑：** 将高频要求词与用户画像的技术栈对比，标记差异项。匹配为语义匹配（如"K8s"视为匹配"Kubernetes"，"Golang"视为匹配"Go"）。

**注意：** 仅采集和分析，不做推荐。

**产出检查点：** 保存为 `job-switch-planner/user_journey.json` 的 `job_listings` 字段。

### CP7：跳槽计划

**目标：** 基于 gap_analysis，生成分阶段跳槽计划。

**输出结构：**

```yaml
switch_plan:
  目标: "云原生 Go 后端工程师"
  时间线: "3个月"
  阶段:
    - 阶段: "第1个月 — 补齐短板"
      学习内容: []
      练习项目: ""
    - 阶段: "第2个月 — 项目包装 + 简历优化"
      行动: []
    - 阶段: "第3个月 — 面试冲刺"
      行动: []
```

**生成规则：**
- 根据 gap 数量确定时间线长度：1-2 个 gap → 1 个月，3-4 个 → 2 个月，5+ 个 → 3 个月
- 每个月聚焦一个主题
- 学习内容包含具体资源链接（来自 LLM 已知知识，如知名书籍、官方文档、经典教程）
- 练习项目贴合实际场景

**交互：** 用户可调整时间线、增减内容。可以进一步生成周计划。

**产出：** 保存到 `job-switch-planner/user_journey.json` 的 `switch_plan` 字段。

## 检查点回退机制

### 存储格式

所有检查点数据集中存储在 `job-switch-planner/user_journey.json`：

```json
{
  "profile": { ... },
  "expectations": { ... },
  "public_opinion": { ... },
  "market_analysis": { ... },
  "recommendations": { ... },
  "selected_positions": [...],
  "job_listings": { ... },
  "switch_plan": { ... }
}
```

### 回退流程

1. 用户在任意阶段结束后说"我想回到 X 阶段重新来"
2. 读取 `job-switch-planner/user_journey.json` 中对应检查点的数据
3. 从该阶段重新开始执行（复用上次输入作为默认值，用户可修改）
4. 该检查点之后的所有数据标记为"过时"
5. 后续阶段按新数据重新计算

### 回退触发条件

- 用户主动要求回退
- 用户在某个阶段修改了关键输入（如改了目标岗位），技能自动建议回退到 CP3 重新分析

## 信息采集（基于 OpenCLI）

使用 OpenCLI（https://github.com/jackwener/OpenCLI）作为采集引擎，提供 130+ 站点的 CLI 适配器。

### 安装

```bash
# 自动安装（推荐）
bash scripts/setup.sh

# 或手动安装
npm install -g @jackwener/opencli

# 首次使用需要先完成登录认证
opencli zhihu search "test"   # 会自动打开浏览器引导登录
```

### 平台支持

| 平台 | OpenCLI 命令 | 认证 | 采集器函数 |
|------|-------------|------|-----------|
| 知乎 | `opencli zhihu search <keyword>` | 需登录 | `collect_zhihu()` |
| Boss直聘 | `opencli boss search <keyword> --city <city>` | 需登录 | `collect_boss()` |
| 前程无忧 | `opencli 51job search <keyword>` | 需登录 | `collect_51job()` |
| Reddit | `opencli reddit search <keyword>` | 需登录 | `collect_reddit()` |
| V2EX | `opencli v2ex hot / latest` | 公开 | `collect_v2ex()` |
| HackerNews | `opencli hackernews search <keyword>` | 公开 | `collect_hackernews()` |

### 采集脚本

`scripts/collector.py` 封装了 OpenCLI 调用，输出统一 JSON：

```bash
python scripts/collector.py --keyword "Go 后端" --output result.json
python scripts/collector.py --keyword "Go 后端" --platforms zhihu,boss,51job
python scripts/collector.py --keyword "golang career" --platforms reddit,hackernews
```

**AI 调用方式：**

```python
import subprocess, json
result = subprocess.run(
    ["python", "scripts/collector.py", "--keyword", "Go 后端", "--output", "/tmp/jobs.json"],
    capture_output=True, text=True
)
data = json.loads(open("/tmp/jobs.json").read())
```

### 采集策略

1. 优先尝试 OpenCLI 自动化采集
2. 如果平台需要登录，引导用户先执行 `opencli <platform> <cmd>` 完成首次认证
3. 如果 OpenCLI 不支持某平台（如猎聘、脉脉），引导用户手动搜索后粘贴给 AI

## 异常处理

### 平台采集异常

| 异常类型 | 表现 | 处理方式 |
|---------|------|---------|
| 未登录 | OpenCLI 返回 auth 错误 | 提示用户执行 `opencli <platform> <cmd>` 完成登录认证 |
| OpenCLI 未安装 | `opencli: command not found` | 提示执行 `npm install -g @jackwener/opencli` |
| 超时 | 命令 30 秒未返回 | 跳过该平台，继续下一平台 |
| 空结果 | 搜索无结果 | 尝试同义词搜索（如"Go后端"→"Golang"） |
| OpenCLI 不支持 | 某平台（如猎聘、脉脉）无适配器 | 跳过，引导用户手动粘贴信息 |

**通用兜底：** 如果所有平台均采集失败，提示用户手动粘贴相关讨论或招聘信息，技能继续完成后续分析。

### 用户行为异常

| 场景 | 处理方式 |
|------|---------|
| 用户中途说"算了不跳了" | 确认后保存当前进度到 `job-switch-planner/user_journey.json`，告知"随时可以回来继续" |
| 用户跳过某个阶段 | 允许跳过 CP3-CP4（公众评价和前景分析），后续推荐使用默认市场数据 |
| 用户反复修改画像 | 每次修改后询问"还有其他要改的吗？"，确认后再继续 |
| 用户提供的信息太少 | 至少需要技术栈 + 工作年限才能继续。不足时说明"目前信息还不足以做分析，至少需要..." |
| 用户偏离话题 | 温和引导回流程："这个我们稍后再聊，先继续当前阶段好吗？" |

### 数据不一致

| 场景 | 处理方式 |
|------|---------|
| 用户回退后修改了画像，但期望没变 | 正常重新计算 CP3 之后的所有阶段 |
| 用户修改了期望岗位 | 建议回退到 CP3 重新采集公众评价和招聘信息 |
| 用户修改了技术栈（增删技能） | CP5-CP7 自动重新计算，CP3-CP4 保持不变 |
| 用户手动编辑了 `job-switch-planner/user_journey.json` | 读取时做字段存在性校验，缺失字段按"未提供"处理 |

### 上下文丢失（对话超时/新会话）

- 每次阶段产出检查点时同步保存到 `job-switch-planner/user_journey.json`
- 新会话启动时检测文件是否存在
- 如果存在，问用户："检测到你有进行中的跳槽分析（上次在 X 阶段），要继续还是重新开始？"
- 重新开始自动备份旧文件为 `user_journey.json.bak`
